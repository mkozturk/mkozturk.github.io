Project structuring for the minimalist data scientist - 1

How to begin setting up your data science projects.
data science software
Author

Kaan Öztürk

Published

February 3, 2024

You are a data scientist beginning with a new task. You think it through, decide on the models you’re going to try, establish your data sources. You are ready to start coding.

How do you start?

Maybe you launch a Jupyter notebook, import your libraries, and start hacking away. After you complete the task, you pass the huge notebook to someone else to put into the so-called “production”. There’s still a lot more to be done before the program suitable for being part of the daily operations, but another poor soul will take care of that.

Even if you can get away with this approach, you may still have some organizational problems. A single folder of notebooks becomes bloated and unmanageable after a short time. It might be difficult to find an old notebook when someone asks you about a an algorithmic detail. When you rerun an old notebook, you frequently get a bunch of error messages, resulting from the changes in the library packages.

After a while, you are tired of this mess. You try to find a better solution. Software engineers have ways of dealing with this complexity. You talk to them, but you cannot make much sense out of their methods, which, on the surface, look like a set of memorized rules (“This is the way!”).

I come from an academic background. Programs I wrote in that career were only for my own consumption, limited to some analysis. Things are different when you write code that will be used by others. So when I switched careers, I had to learn the fundamentals of how to set up a software project. All the difficulties I described above are my experiences.

I took advice from experienced software engineers, but I could not bring myself to follow it verbatim, simply because I could not see the reason for some of it. Without any doubt, they are good advice. But they tend to be maximalist.

“You might need this blahblah.xyz file later, trust me” does not really cut it with me. I need to know the reason of existence for every line of code in my programs. I am a minimalist in coding: I don’t want to keep any unnecessary thing around. (Maybe this has to do with my theoretical science background.)

A case in point is the cookiecutter-datascience template. It is popular among developers and data scientists, and I used in a few projects of mine. It incorporates a lot of good ideas, and I strongly recommend everyone to read and understand it. However, I find this structure difficult to maintain. It has a lot of subdirectories, it is a hassle to navigate through them, and it is not always clear what belongs where.

I’m sure cookiecutter-datascience is very useful once your project reaches a certain level of sophistication. However, for projects I have done so far, a simpler structure turned out to be sufficient.

In a series of posts following this one, I’ll describe how I start a data science project and how I structure its files. I will be writing about virtual environments, how to create installable packages, what to put into your project folder, using GitHub, and how to structure reports and analyses as projects. More importantly, I’ll try to explain why these are necessary. I would always prefer a single file of source code if possible, and I need a good reason to multiply source files.

Tip

Plurality must never be posited without necessity.

— William of Ockham (1495)

None of these will be ironclad principles. I might change my mind about them later. The only strong principle I suggest is: Be a minimalist, use as few things as possible, and know the purpose of everything in your program. Add new things only if you know you need them.

Such minimalism is also a learning opportunity. First, you try to make do as much as possible with your present system. This teaches you the limits of what you can do, and you might discover that you don’t need any extra tool. Second, when you add a new tool into your system, you’ll know exactly why you need it.
