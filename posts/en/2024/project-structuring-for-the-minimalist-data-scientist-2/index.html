In the first part of this series, I described my “minimalist” approach: Be as simple as possible; when you add something, have a good reason for it.

What is a “project”?

I was going to start by saying that you should have a dedicated directory and virtual environment for every data science project you have. But I remembered that I myself could not make sense of this advice when I heard it, because of my own struggle with the concept of “project”. So maybe I should start with a few words about what a project is.

For me, the word “project” had a grand aura around it. A project would take a long time, maybe years. It would involve many people, led by a project manager, which I’m not. My daily work was just writing some programs, producing source files or presentations. So, when my boss asks me to do a forecast of sales, how is that a “project”?

My confusion cleared up when I realized that a project doesn’t have to be as big as a skyscraper or an operating system. In fact, any task with a clear outcome and clear constraints is a project.

So, writing a program to forecast next year’s sales and generating a report is, indeed, a project. Creating a dashboard that queries the database daily and displays the sales graph is another.

You need to be able to say when you are done with a project. So, “researching the applications of LLMs in our market” is not a project, because you can never say that the research is done. When you take up on a task, think about what the outcome should be, and set the boundaries of your project accordingly.

The SMART criteria of goal-setting can be relevant here. They are useful guidelines, but I find them hard to follow to the letter.

In any case, when you have decided on an outcome for your project, you usually know where you are headed, and when it is time to stop. The latter is even more important. Personally, I have a hard time wrapping up projects, so I prioritize defining the stopping criteria.

Project directories

A common advice is to have one directory per project. This usually comes from software engineers, and for good reason. Software projects tend to be self-contained products, and there are usually some auxiliary files that go with them: Source code, config files, credentials, tests, documentation,… I also advise creating one directory for each project, and all the following posts will assume you do.

Sometimes this might look like an unnecessary complication. What if your project is just the analysis of some subset of data? You only produce a Jupyter notebook, nothing more. Creating a dedicated directory for a single notebook is indeed an unnecessary complication, and it makes it difficult to find your files when you need them later. For such tasks, it might be better to collect them under a single directory, say, reports. I describe my approach to that in a later post.
